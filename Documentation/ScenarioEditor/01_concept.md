# 01. コンセプトと業界調査

## ビジョン

**「ライターが頭の中で持っている世界を、そのまま Unity プロジェクトに住まわせる」**

- ノベル/RPG/AVG/オープンワールドなど、シナリオが大規模化するゲームの企画〜実装をひとつの環境で完結させる
- ライター、企画、エンジニア、ローカライザー、声優ディレクションが「同じデータ」を編集視点だけ変えて触れる
- AI による執筆支援/整合性チェック/翻訳が前提。データ構造が AI に解釈しやすい形であることを設計の前提に置く
- Unity Editor 拡張として、ゲーム本体の `ScriptableObject` / プレハブ / シーンに自然に橋渡しする

## 解きたい既存ツールの不満

| 既存ツール | 強み | 不満点 |
|---|---|---|
| **articy:draft 3 / X** | ゲーム業界デファクトのナラティブDB。Flow + Entity + Variable | 高価/有料、ファイル形式独自(XML)、Git でマージ困難、UI重い |
| **Scrivener** | 長編構造管理の老舗。コルクボード、Snapshot、Compile | ゲームのブランチ/状態管理弱い、ロカライズ/エクスポート貧弱 |
| **Final Draft / Fade In / Fountain** | 脚本フォーマット業界標準 | 線形脚本特化。世界設定や分岐は守備範囲外 |
| **World Anvil / Campfire / Plottr** | ワールドビルディング/プロット | Web SaaS が多く、Git 管理に乗らない |
| **Aeon Timeline** | 多軸タイムライン、独自カレンダー | スタンドアロン、ゲーム連携なし |
| **Twine / Ink / Yarn Spinner** | 分岐ダイアログのスクリプト言語 | 設定資料/相関図/年表は別管理になる |
| **Notion / Obsidian** | テキスト柔軟、リンク豊富 | 「ゲーム実装に流す」までの距離が遠い |
| **Excel / Google Sheets** | 結局これに落ち着く現場が多い | バージョン管理/差分追跡が地獄、相関や時間軸の表現は無理 |

→ **「articy の構造化 × Scrivener の長編管理 × Aeon の時間軸 × Yarn の分岐 × Git × Unity」** をひとつにする。

## 業界調査: 本職の現場で実際に使われている機能

シナリオライター・企画ディレクター・ローカライズチーム・QA がそれぞれ求める機能を整理。

### 1. ライティング初期 (世界観設計)

- **エンティティテンプレート** — キャラ/舞台/組織/魔法体系/種族など、型ごとに項目固定
- **相関図** — 親子・所属・敵対・恋愛などのリレーションをグラフ表示
- **時代スナップショット** — 「100年前のXX」「現代のXX」を同一エンティティの Variant として保持
- **モードボード/ムードボード** — 参考画像、配色、雰囲気を貼って共有
- **語彙集 (Glossary)** — 造語・固有名詞・呼称ルール (Mr./San/様 など)
- **語感チェック** — 同じ国の人物名で命名規則統一 (Tolkien方式)

### 2. プロット設計

- **ビートシート** — Save the Cat、Hero's Journey、3幕構成、起承転結、Story Circle
- **シーンカード** — 1シーン=1カードでドラッグ並べ替え (Scrivener コルクボード)
- **POV/視点キャラ** タグ
- **盛り上がりカーブ** — 緊張度を 0–100 で時系列にグラフ化
- **伏線管理** — 仕込み (planted) / 回収 (paid off) 紐付け
- **ループ進行管理** — 同じ場面の差分管理 (周回ゲー、ループもの)

### 3. 脚本/ダイアログ執筆

- **登場人物のサムネ + 名前を行頭に**(ライター要望最頻出)
- **発話の感情タグ** (`@happy` `@angry` 等) — 表情立ち絵/Live2D/lipsync 連動
- **演出指示** — `[SE: 雷鳴]` `[Camera: pan_left]` `[Wait: 1.0s]`
- **分岐 (Choice)** — 選択肢、条件、フラグ操作
- **変数置換** — `{playerName}` `{her_or_him}`
- **行ごとの最大文字数制限** — UI 表示枠/吹き出しに収まるか
- **声優収録メタデータ** — Cue#、ファイル名、最大尺、読みづらい単語の発音メモ
- **Take 2/3 録り直し管理** — 改稿時の収録差分

### 4. 校正・整合性

- **文字数/読了時間/喋り時間推定**
- **キャラ口調の一貫性** — 「私/俺/僕」混在チェック
- **大事な情報の登場順** — 死亡したキャラが後で生きてないか
- **年齢計算** — エポックを跨いだ自動計算
- **継続性 (Continuity)** — 髪色/瞳色/装備品の作品内変化を track

### 5. ローカライズ

- **キー固定の翻訳テーブル** (i18n key)
- **翻訳メモリ (TM)** — 過去訳の流用
- **コンテキスト** — 話者性別/口調/前後文脈/参考画像
- **文字コード/長さ制限/改行ルール** (英→日でフォントメトリクスが変わる)
- **音声差し替えとの整合性**

### 6. 制作管理 (Production)

- **Tagger** — 「このシーンに必要な美術/SE/3Dモデル」を抽出
- **収録台本生成** — キャラ別、シーン別、ファイル別
- **進捗ダッシュボード** — シーン別 Status (Draft/Review/Approved/Recorded)
- **改訂モード** — 色分け差分 (Final Draft の Revisions)

### 7. ゲーム実装連携

- **ScriptableObject にエクスポート** — Unity から直接参照
- **YAML/JSON で持ち出し** — エンジン非依存
- **Yarn / Ink フォーマットで出力** — 既存ランタイム流用
- **Hot-reload** — エディタ実行中の差し替え
- **CI で整合性検査** — 未翻訳キー/未定義参照の検出

## 設計判断のキーポイント

1. **「ノード型」を世界の最小単位にする** — articy の Entity と同じ思想。型はテンプレートで定義可能。
2. **Variant を時代軸で持つ** — Aeon Timeline の Era とは違い、エンティティ自身が時代差分を保持する。
3. **エディタは UIToolkit + 一部 IMGUI** — グラフは UIToolkit GraphView、テーブル系は ListView、レガシな部分は IMGUI。
4. **データはファイルシステム上の YAML** — `ScriptableObject` ではなく素の `.yaml` を一次ソースに。Unity 側は AssetImporter で `ScriptableObject` を生成。
5. **Git ファースト** — 1ノード=1ファイルが基本。マージ衝突を減らすためのファイル分割粒度を意識。
6. **AI フレンドリー** — フィールド名は英字キー＋日本語ラベル併記。スキーマがプロンプトに乗る大きさに収める。
